# 3~4장 통합 구현 / 서버 프로그램 구현 — 예상 문제 (p.32~41)

---

## 유형 1. 설명 → 용어 맞추기

> 설명을 읽고 해당하는 용어를 쓰시오.

**Q01.**  
특수한 목적을 갖는 마크업 언어를 만드는 데 사용되는 **다목적 마크업 언어**이다. 사용자가 직접 태그(Tag)를 정의할 수 있으며, 트리 구조로 구성되어 상위 태그는 여러 개의 하위 태그를 가질 수 있다.

**Q02.**  
컴퓨터 네트워크 상에서 HTTP/HTTPS, SMTP 등을 이용하여 **XML을 교환하기 위한 통신 규약**이다. 기본적으로 HTTP 기반에서 동작하기 때문에 프록시와 방화벽의 영향 없이 통신할 수 있다.

**Q03.**  
웹 서비스와 관련된 서식이나 프로토콜 등을 **표준적인 방법으로 기술하고 게시하기 위한 언어**이다. XML로 작성되며, UDDI의 기초가 된다.

**Q04.**  
소프트웨어의 성능 향상, 시스템의 수정 및 재사용, 유지 관리 등이 용이하도록 시스템의 기능들을 **모듈 단위로 나누는 것**을 의미한다. 모듈 간 **결합도(Coupling) 최소화**와 **응집도(Cohesion) 최대화**가 목표이다.

**Q05.**  
아키텍처 패턴 중, **데이터 스트림 절차의 각 단계를 필터로 캡슐화하여 파이프를 통해 전송**하는 패턴이다. 데이터 변환, 버퍼링, 동기화에 주로 사용되며, UNIX의 쉘(Shell)이 대표 예이다.

**Q06.**  
아키텍처 패턴 중, **사용자가 원하는 서비스를 브로커 컴포넌트에 요청하면**, 브로커가 요청에 맞는 컴포넌트와 사용자를 연결해주는 패턴이다. 분산 환경 시스템에 사용된다.

**Q07.**  
객체지향 개념 중, **공통된 속성과 연산을 갖는 객체의 집합**이다. 각각의 객체들이 갖는 속성과 연산을 정의하고 있는 틀이며, 여기에 속한 각각의 객체를 인스턴스(Instance)라고 한다.

**Q08.**  
객체지향 개념 중, **외부에서의 접근을 제한하기 위해 인터페이스를 제외한 세부 내용을 은닉**하는 것이다. 이를 통해 객체들 간의 결합도가 낮아지고 인터페이스가 단순해진다.

**Q09.**  
모든 소프트웨어 구성 요소를 그래픽 표기법으로 모델링하는 기법이다. **객체 모델링 → 동적 모델링 → 기능 모델링** 순으로 분석 활동이 이루어지며, OMT(Object Modeling Technique)라고도 한다.

**Q10.**  
SOLID 원칙 중, **객체는 단 하나의 책임만 가져야 한다**는 원칙이다.

**Q11.**  
SOLID 원칙 중, **기존의 코드를 변경하지 않고 기능을 추가**할 수 있도록 설계해야 한다는 원칙이다.

**Q12.**  
SOLID 원칙 중, **의존 관계 성립 시 추상성이 높은 클래스와 의존 관계를 맺어야 한다**는 원칙이다.

**Q13.**  
모듈 간에 상호 의존하는 정도로, **약할수록 품질이 높고 강할수록 품질이 낮다**. 내용 > 공통 > 외부 > 제어 > 스탬프 > 자료 결합도 순으로 강도가 약해진다.

**Q14.**  
결합도의 종류 중, **한 모듈이 다른 모듈의 내부 기능 및 그 내부 자료를 직접 참조하거나 수정**할 때의 결합도이다. 결합도 중 가장 강하다.

**Q15.**  
결합도의 종류 중, **모듈 간의 인터페이스가 자료 요소로만 구성**될 때의 결합도이다. 결합도 중 가장 약하다.

**Q16.**  
모듈의 내부 요소들이 서로 관련되어 있는 정도이다. **강할수록 품질이 높고 약할수록 품질이 낮다**. 기능 > 순차 > 교환 > 절차 > 시간 > 논리 > 우연 응집도 순으로 강도가 약해진다.

**Q17.**  
응집도의 종류 중, **모듈 내부의 모든 기능 요소들이 단일 문제와 연관되어 수행**될 경우의 응집도이다. 가장 강한 응집도이다.

**Q18.**  
응집도의 종류 중, **모듈 내부의 각 구성 요소들이 서로 관련 없는 요소로만 구성**된 경우의 응집도이다. 가장 약한 응집도이다.

**Q19.**  
어떤 모듈을 **제어하는 모듈의 수**를 의미한다. (어떤 모듈이 몇 개의 모듈에 의해 호출되는가)

**Q20.**  
어떤 모듈에 의해 **제어되는 모듈의 수**를 의미한다. (어떤 모듈이 몇 개의 모듈을 호출하는가)

**Q21.**  
**모듈 간 통신 방식을 구현**하기 위해 사용되는 대표적인 프로그래밍 인터페이스 집합이다. 공유 메모리, 소켓, 세마포어, 파이프, 메시지 큐잉 등 5가지 대표 메소드가 있다.

**Q22.**  
구현된 소프트웨어가 사용자의 요구사항을 정확하게 준수했는지를 확인하기 위한 **테스트 항목에 대한 명세서**이다. 식별자, 테스트 항목, 입력 명세, 출력 명세, 환경 설정 등으로 구성된다.

**Q23.**  
이미 개발된 기능들을 새로운 시스템이나 기능 개발에 사용하기 적합하도록 **최적화하는 작업**이다. 함수와 객체 / 컴포넌트 / 애플리케이션 단위로 분류된다.

**Q24.**  
모듈 간의 관계 및 인터페이스를 설계할 때 참조할 수 있는 **전형적인 해결 방식 또는 예제**이다. GOF의 패턴은 생성 패턴, 구조 패턴, 행위 패턴으로 구분된다.

**Q25.**  
디자인 패턴의 생성 패턴 중, **클래스 내에서 인스턴스가 하나뿐임을 보장**하며, 생성된 객체를 어디서든 참조할 수 있는 패턴이다.

**Q26.**  
디자인 패턴의 생성 패턴 중, **객체 생성을 서브 클래스에서 처리하도록 분리하여 캡슐화**한 패턴이다. 상위 클래스는 인터페이스만 정의하고 실제 생성은 서브 클래스가 담당한다. 가상 생성자 패턴이라고도 한다.

**Q27.**  
디자인 패턴의 구조 패턴 중, **호환성이 없는 클래스들의 인터페이스를 다른 클래스가 이용할 수 있도록 변환**해주는 패턴이다.

**Q28.**  
디자인 패턴의 구조 패턴 중, **복잡한 서브 클래스들을 피해 더 상위에 인터페이스를 구성**함으로써 서브 클래스들의 기능을 간편하게 사용할 수 있도록 하는 패턴이다.

**Q29.**  
디자인 패턴의 행위 패턴 중, **한 객체의 상태가 변화하면 상속되어 있는 다른 객체들에게 변화된 상태를 전달**하는 패턴이다. 일대다의 의존성을 정의하며, Publish/Subscribe 방식에 사용된다.

**Q30.**  
디자인 패턴의 행위 패턴 중, **동일한 계열의 알고리즘들을 개별적으로 캡슐화하여 상호 교환할 수 있게 정의**하는 패턴이다. 클라이언트에 영향 없이 알고리즘 변경이 가능하다.

## 유형 2. 핵심 빈칸 뚫기

**Q31.**  
럼바우(Rumbaugh)의 분석 기법은 분석 활동을 세 가지 모델로 나누어 수행한다. 빈칸을 채우시오.

분석 순서 : ( ① ) 모델링 → ( ② ) 모델링 → ( ③ ) 모델링

각 모델링의 특징:
- ( ① ) 모델링: 객체를 찾아내어 속성과 연산 식별, 객체 다이어그램으로 표시
- ( ② ) 모델링: 상태 다이어그램 이용, 시간의 흐름에 따른 동적 행위 표현
- ( ③ ) 모델링: 자료 흐름도(DFD) 이용, 다수의 프로세스들 간 자료 흐름 표현

**Q32.**

① SRP : ( ㄱ ) 원칙

③ LSP : ( ㄷ ) 원칙  
④ ISP : ( ㄹ ) 원칙  
⑤ DIP : ( ㅁ ) 원칙

**Q34.**  
응집도의 종류를 강한 것부터 약한 순으로 나열하시오.

( ① ) > ( ② ) > ( ③ ) > ( ④ ) > ( ⑤ ) > ( ⑥ ) > ( ⑦ )

IPC의 대표 메소드 5가지를 쓰시오.
